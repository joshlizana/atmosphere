The `postgres` service uses image `postgres:17-bookworm`. Its env block maps the official image's expected superuser pair (`POSTGRES_USER` / `POSTGRES_PASSWORD`) plus the re-declared `POSTGRES_{SUPERUSER,LAKEKEEPER,PREFECT,OPENMETADATA,GRAFANA}_{USER,PASSWORD}` keys sourced from `.env`. The host port is bound `5432:5432`, the healthcheck runs `CMD-SHELL pg_isready -U $${POSTGRES_SUPERUSER} -d postgres` on a 10s interval with 10 retries and a 30s start_period, the soft memory floor is `mem_reservation: 768m`, and `restart: unless-stopped` is in effect. ^p002

The `seaweedfs` service uses image `chrislusf/seaweedfs:4.21` with command `server -dir=/data -filer -s3 -master.defaultReplication=000 -metricsPort=9327`. The key architectural decision encoded here is the deliberate omission of `-s3.config` — per the JDL-66 decision recorded in `[[decisions/JDL-66/20260424T1236Z-seaweedfs-s3-credentials-via-env-vars]]`, the admin identity synthesizes from `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` env vars (mapped from `SEAWEEDFS_S3_{ACCESS,SECRET}_KEY` in `.env.example`), plus `WEED_CLUSTER_DEFAULT_REPLICATION=000` as defense-in-depth for the CLI flag. The bind mount is `./data/seaweedfs:/data`; ports are 9333 (master), 8080 (volume), 8888 (filer), 8333 (S3), and 9327 (Prometheus metrics). Healthcheck is `CMD-SHELL wget --spider -q http://localhost:9333/cluster/status || exit 1` on a 10s interval with 6 retries and a 20s start_period. Soft `mem_reservation: 1g`, `restart: unless-stopped`. ^p003

The `seaweedfs-init` service is the bucket-provisioning one-shot — same image as `seaweedfs`, with `depends_on: { seaweedfs: { condition: service_healthy } }` so it runs only after the main container is live. Its `entrypoint: ["/bin/sh", "-c"]` pipes a `printf 's3.bucket.create -name %s\n'` over the three bucket names into `weed shell -master=seaweedfs:9333`, creating the platform's three buckets (`atmosphere`, `flink`, `loki`). The bucket names are taken from `SEAWEEDFS_BUCKET_{ATMOSPHERE,FLINK,LOKI}` env vars so a future rename is a `.env` change rather than a compose edit, and `restart: "no"` ensures the container exits 0 and stays down. `weed shell`'s `s3.bucket.create` is idempotent by design, so the step is safe to re-run on every bring-up without guard logic. ^p004

The `redpanda` service uses image `docker.redpanda.com/redpandadata/redpanda:v26.1.6`. Its command carries the single-broker flag set: `--smp=1 --memory=1536M --reserve-memory=0M --overprovisioned --node-id=0 --check=false --kafka-addr=PLAINTEXT://0.0.0.0:9092 --advertise-kafka-addr=PLAINTEXT://redpanda:9092 --rpc-addr=redpanda:33145 --advertise-rpc-addr=redpanda:33145 --seeds=redpanda:33145 --admin-addr=0.0.0.0:9644`. Key omissions — encoded in a service-header comment rather than in explicit flags because there is no explicit "off" switch for these subsystems — are `--mode dev-container` (disables fsync, a data-loss footgun), `--schema-registry-addr`, and `--pandaproxy-addr` (their absence is how those subsystems stay disabled; the default schema-registry port 8081 would collide with Flink's JobManager UI in M5, and the platform uses neither subsystem). Volumes are `./data/redpanda:/var/lib/redpanda/data` for broker state and `./config/redpanda/.bootstrap.yaml:/etc/redpanda/.bootstrap.yaml:ro` for the cluster-property seed documented in `[[files/JDL-66/config/redpanda/bootstrap-yaml]]`. Ports 9092 (Kafka) and 9644 (admin) are exposed; RPC port 33145 is intentionally kept internal. Healthcheck is `CMD-SHELL curl -sf http://localhost:9644/v1/status/ready || exit 1` on a 10s interval with 10 retries and a 30s start_period. Soft `mem_reservation: 1.5g`, which agrees numerically with `--memory=1536M` so the container's declared memory budget and Docker's reservation are not in tension, and `restart: unless-stopped`. ^p005

The top-level `volumes:` block declares `atmosphere_postgres:` with an explicit `name: atmosphere_postgres` override — a fix for a subtle Compose-prefix gotcha Forge caught during implementation. Compose prefixes top-level volume keys with `COMPOSE_PROJECT_NAME`, so the naive form would yield `atmosphere_atmosphere_postgres` on disk; the `name:` override pins the literal `atmosphere_postgres` identifier that every backup/admin reference in `.claude/context/operations.md` expects. `name:` is not a driver option, so the design doc's "no driver options" invariant still holds — the override is a naming pin, not a driver customization. ^p006

The file has been linted via `prettier --check compose.yml` and passes cleanly. `docker compose config` parses successfully, emitting only the expected `variable is not set` warnings for operator-supplied `.env` values — the canonical green-light signal for the M1 scaffolding stage before `.env` is populated. ^p007

Fix 1 — Redpanda admin listener. In the Redpanda service's `command:` list, replaced the single line `- --admin-addr=0.0.0.0:9644` with two lines: `- --set=redpanda.admin[0].address=0.0.0.0` and `- --set=redpanda.admin[0].port=9644`. Root cause of the prior failure: `--admin-addr` is not a valid `rpk redpanda start` flag in `docker.redpanda.com/redpandadata/redpanda:v26.1.6` — the Seastar broker binary rejects it with `Argument parse error: unrecognised option`, and the container crash-loops. The `--set redpanda.admin[N].address/port` form uses the `parseConfigKvs` function in `src/go/rpk/pkg/cli/redpanda/start.go` at the v26.1.6 tag to inject node-scoped properties through the same path `rpk redpanda config set` uses. Critical detail: the admin listener default in v26.1.x is `127.0.0.1:9644` (not `0.0.0.0`), so dropping the flag without replacement would have swapped the crash-loop for a silently-unreachable admin port (host port-map to a container-loopback listener would fail). Scout research at `[[research/JDL-66/redpanda-admin-api-listener-v26]]` (commit `e2d950d`) pinned this as the canonical v26.1.x pattern. ^p014

Fix 2 — SeaweedFS healthcheck. In the SeaweedFS service's `healthcheck:` block, replaced the test array with `["CMD-SHELL", "wget -q -O /dev/null http://127.0.0.1:9333/cluster/healthz || exit 1"]`. Three orthogonal changes bundled, each an independent defense against the BusyBox-wget gotcha class: (1) literal IPv4 `127.0.0.1` instead of `localhost` — the Alpine 3.23 image's `/etc/hosts` resolves `localhost` to `::1` first, and SeaweedFS binds IPv4 loopback only, so BusyBox wget was connection-refused on the IPv6 record; (2) `/cluster/healthz` instead of `/cluster/status` — the dedicated Kubernetes-style liveness endpoint upstream added in SeaweedFS PR #2680, returns 200 empty body; (3) `wget -q -O /dev/null` instead of `wget --spider -q` — avoids a separate BusyBox bug where `--spider` returns exit 0 on "can't connect" under some flag combinations. Scout research at `[[research/JDL-66/seaweedfs-healthcheck-busybox-wget-2026]]` (commit `9db0e56`) live-verified each step against the `chrislusf/seaweedfs:4.21` image. Interval/timeout/retries/start_period (10s/5s/6/20s) kept unchanged. ^p015

Everything else in `compose.yml` preserved verbatim: postgres service, seaweedfs service command / env / volumes / ports / mem_reservation / restart, seaweedfs-init service, top-level `volumes:` block, every other Redpanda flag (`--smp=1`, `--memory=1536M`, `--reserve-memory=0M`, `--overprovisioned`, `--node-id=0`, `--check=false`, `--kafka-addr`, `--advertise-kafka-addr`, `--rpc-addr`, `--advertise-rpc-addr`, `--seeds`). ^p016

Three localized changes in the Redpanda service block, all to work around `rpk`'s insistence on chown-ing a temp file in the config directory at startup — which fails against a read-only bind mount. ^p023

Change 1 — added `entrypoint:` wrapper. The new list is `/bin/sh -c 'cp /run/config/redpanda.yaml /etc/redpanda/redpanda.yaml && exec "$@"' --`. The shell copies the mounted source at container start into `rpk`'s default config path (`/etc/redpanda/redpanda.yaml`) — a fully writable tmpfs-backed location inside the container — then `exec "$@"` hands control to the `command:` list. `rpk`'s temp-file-chown-rename dance now operates on the copy, never touching the host source. The `--` sentinel at the end of the entrypoint list is the standard Docker wrapper idiom — it becomes `$0` in the shell so `$@` starts at the first word of `command:`. ^p024

Change 2 — prepended `rpk` to `command:`. The list previously started `- redpanda` / `- start`. Now starts `- rpk` / `- redpanda` / `- start`. This ensures the shell's `exec "$@"` produces the full `rpk redpanda start …` invocation. All other command flags (`--smp=1`, `--memory=1536M`, `--reserve-memory=0M`, `--overprovisioned`, `--node-id=0`, `--check=false`, `--kafka-addr=PLAINTEXT://0.0.0.0:9092`, `--advertise-kafka-addr=PLAINTEXT://redpanda:9092`, `--rpc-addr=redpanda:33145`, `--advertise-rpc-addr=redpanda:33145`, `--seeds=redpanda:33145`) preserved verbatim in order. ^p025

Change 3 — changed the volume mount target. The `config/redpanda/redpanda.yaml` source now mounts at `/run/config/redpanda.yaml:ro` instead of `/etc/redpanda/redpanda.yaml:ro`. The entrypoint's `cp` copies from there into the default `rpk` path. The `.bootstrap.yaml` mount at `/etc/redpanda/.bootstrap.yaml:ro` stays untouched; the `atmosphere_redpanda` named volume at `/var/lib/redpanda/data` stays untouched. ^p026

Forge-verified via upstream image inspection. Redpanda's v26.1.6 image ships `/entrypoint.sh` as a thin bash wrapper gated on `ENABLE_DEFAULT_LISTENERS`; with that env unset (our case), the wrapper's else branch is just `exec /usr/bin/rpk $@`. Overriding entrypoint with the cp-wrapper loses no signal handling or privilege dropping — safe one-for-one replacement. ^p027

Forge-surfaced gotcha. `docker compose config` renders `"$@"` in the entrypoint as `"$$@"` in its output — display-escaping for round-trip safety, not a runtime behavior. The written YAML uses `"$@"` correctly; rewriting to `"$$@"` would break the exec. Forge saved this as a memory entry for future compose-entrypoint work. ^p028

Every other Redpanda service field (`image`, `environment`, `ports`, `healthcheck`, `mem_reservation`, `restart`) preserved verbatim. Every other compose service (`postgres`, `seaweedfs`, `seaweedfs-init`) preserved verbatim. Top-level `volumes:` block preserved verbatim. ^p029
