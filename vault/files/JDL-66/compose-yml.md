Forge modified `compose.yml` at commit `bb4fc7c` on branch `JDL-66-m1-shared-infrastructure-online`. The file previously held only a header comment block, `name: atmosphere`, and an empty `services: {}` map (the M0 / JDL-65 landing recorded in `[[files/JDL-65/compose-yml]]`). The modification replaces the stub with four service definitions and a top-level `volumes:` declaration, taking the compose manifest from "parses cleanly with no services" to the first functional M1 infrastructure topology. ^p001

The `postgres` service uses image `postgres:17-bookworm`. Its env block maps the official image's expected superuser pair (`POSTGRES_USER` / `POSTGRES_PASSWORD`) plus the re-declared `POSTGRES_{SUPERUSER,LAKEKEEPER,PREFECT,OPENMETADATA,GRAFANA}_{USER,PASSWORD}` keys sourced from `.env`. Volumes are the docker-managed `atmosphere_postgres` named volume mounted at `/var/lib/postgresql/data` plus `./config/postgres` read-only at `/docker-entrypoint-initdb.d` for the init SQL. The host port is bound `5432:5432`, the healthcheck runs `CMD-SHELL pg_isready -U $${POSTGRES_SUPERUSER} -d postgres` on a 10s interval with 10 retries and a 30s start_period, the soft memory floor is `mem_reservation: 768m`, and `restart: unless-stopped` is in effect. ^p002

The `seaweedfs` service uses image `chrislusf/seaweedfs:4.21` with command `server -dir=/data -filer -s3 -master.defaultReplication=000 -metricsPort=9327`. The key architectural decision encoded here is the deliberate omission of `-s3.config` — per the JDL-66 decision recorded in `[[decisions/JDL-66/20260424T1236Z-seaweedfs-s3-credentials-via-env-vars]]`, the admin identity synthesizes from `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` env vars (mapped from `SEAWEEDFS_S3_{ACCESS,SECRET}_KEY` in `.env.example`), plus `WEED_CLUSTER_DEFAULT_REPLICATION=000` as defense-in-depth for the CLI flag. The bind mount is `./data/seaweedfs:/data`; ports are 9333 (master), 8080 (volume), 8888 (filer), 8333 (S3), and 9327 (Prometheus metrics). Healthcheck is `CMD-SHELL wget --spider -q http://localhost:9333/cluster/status || exit 1` on a 10s interval with 6 retries and a 20s start_period. Soft `mem_reservation: 1g`, `restart: unless-stopped`. ^p003

The `seaweedfs-init` service is the bucket-provisioning one-shot — same image as `seaweedfs`, with `depends_on: { seaweedfs: { condition: service_healthy } }` so it runs only after the main container is live. Its `entrypoint: ["/bin/sh", "-c"]` pipes a `printf 's3.bucket.create -name %s\n'` over the three bucket names into `weed shell -master=seaweedfs:9333`, creating the platform's three buckets (`atmosphere`, `flink`, `loki`). The bucket names are taken from `SEAWEEDFS_BUCKET_{ATMOSPHERE,FLINK,LOKI}` env vars so a future rename is a `.env` change rather than a compose edit, and `restart: "no"` ensures the container exits 0 and stays down. `weed shell`'s `s3.bucket.create` is idempotent by design, so the step is safe to re-run on every bring-up without guard logic. ^p004

The `redpanda` service uses image `docker.redpanda.com/redpandadata/redpanda:v26.1.6`. Its command carries the single-broker flag set: `--smp=1 --memory=1536M --reserve-memory=0M --overprovisioned --node-id=0 --check=false --kafka-addr=PLAINTEXT://0.0.0.0:9092 --advertise-kafka-addr=PLAINTEXT://redpanda:9092 --rpc-addr=redpanda:33145 --advertise-rpc-addr=redpanda:33145 --seeds=redpanda:33145 --admin-addr=0.0.0.0:9644`. Key omissions — encoded in a service-header comment rather than in explicit flags because there is no explicit "off" switch for these subsystems — are `--mode dev-container` (disables fsync, a data-loss footgun), `--schema-registry-addr`, and `--pandaproxy-addr` (their absence is how those subsystems stay disabled; the default schema-registry port 8081 would collide with Flink's JobManager UI in M5, and the platform uses neither subsystem). Volumes are `./data/redpanda:/var/lib/redpanda/data` for broker state and `./config/redpanda/.bootstrap.yaml:/etc/redpanda/.bootstrap.yaml:ro` for the cluster-property seed documented in `[[files/JDL-66/config/redpanda/bootstrap-yaml]]`. Ports 9092 (Kafka) and 9644 (admin) are exposed; RPC port 33145 is intentionally kept internal. Healthcheck is `CMD-SHELL curl -sf http://localhost:9644/v1/status/ready || exit 1` on a 10s interval with 10 retries and a 30s start_period. Soft `mem_reservation: 1.5g`, which agrees numerically with `--memory=1536M` so the container's declared memory budget and Docker's reservation are not in tension, and `restart: unless-stopped`. ^p005

The top-level `volumes:` block declares `atmosphere_postgres:` with an explicit `name: atmosphere_postgres` override — a fix for a subtle Compose-prefix gotcha Forge caught during implementation. Compose prefixes top-level volume keys with `COMPOSE_PROJECT_NAME`, so the naive form would yield `atmosphere_atmosphere_postgres` on disk; the `name:` override pins the literal `atmosphere_postgres` identifier that every backup/admin reference in `.claude/context/operations.md` expects. `name:` is not a driver option, so the design doc's "no driver options" invariant still holds — the override is a naming pin, not a driver customization. ^p006

The file has been linted via `prettier --check compose.yml` and passes cleanly. `docker compose config` parses successfully, emitting only the expected `variable is not set` warnings for operator-supplied `.env` values — the canonical green-light signal for the M1 scaffolding stage before `.env` is populated. ^p007

Forge modified `compose.yml` at commit `bee21ab` on branch `JDL-66-m1-shared-infrastructure-online` (+16/-8). The change converts the two remaining host-bind-mount state paths to docker-managed named volumes, finishing the architectural shift to uniform volume management across every stateful M1 service. After this landing, no M1 service persists state via a host bind mount. ^p008

The `seaweedfs.volumes` entry flipped from `./data/seaweedfs:/data` to `atmosphere_seaweedfs:/data`, and `redpanda.volumes` flipped from `./data/redpanda:/var/lib/redpanda/data` to `atmosphere_redpanda:/var/lib/redpanda/data`. The Redpanda config-file bind mount `./config/redpanda/.bootstrap.yaml:/etc/redpanda/.bootstrap.yaml:ro` stayed as-is — it's an immutable-input mount for a read-only config file, which is a distinct sub-pattern from state persistence and falls outside the scope of this decision (see `[[decisions/JDL-66/20260424T1518Z-docker-named-volumes-for-state]]` ^p001). ^p009

The top-level `volumes:` block gained two new entries — `atmosphere_seaweedfs` and `atmosphere_redpanda` — each with a `name: <literal>` override mirroring the existing `atmosphere_postgres` pattern recorded in the initial entry above (^p006). The override pins the literal on-disk volume name regardless of `COMPOSE_PROJECT_NAME`; without it Compose would prefix the key and yield `atmosphere_atmosphere_seaweedfs` on disk, breaking the backup-enumeration contract (`docker volume ls --filter name=atmosphere_`) that the broader operations model depends on. The existing `atmosphere_postgres:` entry was untouched. ^p010

The volume-level comment block was generalized from its Postgres-specific prose to describe the named-volume-plus-`name:`-override convention as the platform-wide pattern for stateful services. This matches the forward posture declared in `[[decisions/JDL-66/20260424T1518Z-docker-named-volumes-for-state]]` (every new stateful service gets an `atmosphere_<service>` docker-managed named volume with a `name:` override) so the comment now reads as platform convention rather than one-off justification. ^p011

Validation. `docker compose config --quiet` parses clean; `npx prettier --check compose.yml` passes. This landing, combined with the earlier `scripts/up.sh` drop of the `mkdir` + `sudo chown` preflight recorded in `[[files/JDL-66/scripts/up-sh]]`, completes the operator-loop simplification: `docker volume ls --filter name=atmosphere_` is now the single enumeration path for platform state, and there is no remaining asymmetry between Postgres and the other stateful services. ^p012
